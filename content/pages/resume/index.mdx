## 자기소개

### 명확한 기준과 문서화를 추구하는 프로그래머

처음으로 팀원과 협업했던 당시 체계적인 단계를 거쳐 개발하거나 명확한 기준과 문서를 기반을 두어 작업을 하지 않았습니다. 서로에게 명확하지 않은 요청을 하며 개발을 진행했습니다. 요청이 이해가 되지 않는 경우가 발생하여 개발이 멈추고, 테스트할 때마다 원하는 결과가 나오지 않았습니다. 다음 프로젝트에서는 기능을 설계할 때 문서화를 잘하고, 주기적인 회의를 거쳐 기록하기로 했습니다. 다음 회의까지 기능을 구현하고, 진척도와 새로운 이슈를 준비하도록 하였습니다. 문서화와 주기적인 회의로 각 부분 담당자들의 개발 진척도는 다른 프로젝트보다 빨라졌고, 테스트와 시연 시 실패 확률도 줄어들었습니다. 이러한 문서 관리와 진척도 관리를 해주는 솔루션을 이용하여 팀원 모두가 개발에 집중할 수 있도록 하였습니다.

### 개발에서 최종 배포까지 염두 하는 프로그래머

아무리 좋은 프로그램 또는 서비스라도 최종 사용자나 관리자가 설치하기 어려우면 그것은 정적인 문서이고 어떠한 결과도 내놓지 못합니다. 또한, 설치 용이성은 사용자에게 주는 첫 사용자 경험이기에 놓칠 수 없습니다. 보다 더 많은 사용자와 개발자들에게 다가가기 위해 배포하고자 하는 플랫폼을 늘리기 위해 크로스플랫폼을 고려하거나 운영체제 독립적인 코드 작성 또는 배포 방식을 고민해보았습니다. 공식적인 스토어나 커뮤니티 허브에 배포하기 위한 노력 또한 해보았습니다. 배포 환경의 보안규정에 맞추어 전자서명을 위한 인증서를 발급하거나 COC에 서명을 하는 등의 일련의 행위를 하여 패키지의 신뢰성을 높이는 작업을 하였습니다. 배포처의 기능과 연계하여 새로운 기능이나 버그 수정 버전이 나오면 사용자에게 알림을 주도록 하였습니다.

### Qiskit-classroom

개발 시기 : 2023년 (3인 개발)

사용 기술 : Python, Qt

- 양자컴퓨팅에서 사용되는 표현식을 상호변환하고, 시각화하는 GUI 애플리케이션 개발
- Qiskit ecosystem에 프로젝트가 편입되기 위한 조건 달성
  - Qiskit 패키지와 프로젝트의 호환성을 위해 테스트 코드 추가
  - 내부적으로 coverage를 80%이상 유지하도록 목표함
  - MVP패턴을 도입하여 테스트 코드 작성을 용이하도록 만듦
  - Mocking을 도입하여 system call 호출 시 소요되는 시간 절약
- 오픈소스 라이선스 관리
  - 가장 제약이 적은 Apache 라이선스 도입
  - 외부 패키지 사용 시 Apache 라이선스와 충돌하는지 확인하고 도입
- 애플리케이션의 설치 용이성을 늘리기 위해 노력
  - PyPI와 Debian에 배포 시도
  - 보다 많은 배포판을 지원하기 위해 snap, flatpak을 도입

### ISeeYou

개발 시기 : 2016년 (2인 개발)

사용 기술 : C, Visual Basic 6

- 교육용 프레젠테이션 리모콘 애플리케이션
- PC의 입력 잠금, 전원관리, 프레젠테이션 리모콘 기능을 제공
- 서비스 전용 서버를 두어 다른 네트워크에 존재하는 클라이언트도 제어 가능
- Visual Basic 6을 이용한 개발 도중 UAC와 충돌하여 내부 구현 수정
  - 관리자 기능을 사용하면 애플리케이션이 중지하는 경우 발생
  - 관리자 기능을 사용하지 않고 해당 기능을 구현하는 방식으로 해결
  - UAC 프롬프트를 띄우지 않고 요구 사항을 충족

2017년 (1인 유지 보수)

- 클라이언트의 코드를 바꾸지 않고 서버를 유지 보수하는 것을 목표로 함
  - 리눅스로 서버를 배포해야 되는 경우가 발생하여 전처리문을 이용하여 해결
  - 모든 클라이언트에게 메시지를 브로드캐스트한 것을 유니캐스트로 변경하여 system call 호출 회수를 줄임
- 클라이언트 배포관련 유지보수
  - 설치 시 “신뢰할 수 없는 소프트웨어” 알림과 함께 설치 용이성이 떨어짐
  - Code Signing을 위한 인증서 발급으로 해결 가능하나 Microsoft Store에 배포하는 것이 인증서 비용 절감 및 업데이트 알림 등의 이득이 있음을 확인함
